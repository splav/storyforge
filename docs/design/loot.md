> **СТАТУС: ДРАФТ. Ничего из этого ещё не реализовано.** Фиксирует дизайн-решения
> из обсуждения. Спецификация *намерения*, не описание текущего поведения.
> Числа — иллюстративные, не оттюненные. Парный документ —
> [`progression.md`](progression.md).

---

## 0. Что это решает

Сегодня гир персонажа фиксирован классом (`ClassDef`/шаблон): надетое не меняется.
Цель — **живой лоадаут**, который эволюционирует через дроп + магазин, и который
несёт **всю вертикаль силы** (базовые статы только с гира, см. `progression.md §1`).

---

## 1. Грунтовка в текущих данных

Гир уже несёт **полный стат-словарь**:

- `WeaponDef`: `dice`, `spell_power`, `armor`, `max_hp`, СИЛ/ЛОВ/ТЕЛ/ИНТ, `hand`.
- `ArmorDef`: `armor`, `max_hp`, СИЛ/ЛОВ/ТЕЛ/ИНТ, `slot` (грудь/ноги/стопы).
- Слоты: main/off/two-handed оружие + 3 брони.

Следствия:
- «Стат-профиль» как ось итемизации **уже работает** (посох `+INT` vs меч `+STR`).
- Тяжёлая броня с **`−ловкость`** (из прогрессии) — это просто отрицательное значение в стат-строке, новой системы не надо.

Чего **нет**: рарити / аффиксов / тиров / генерации (каждый предмет — плоский авторский деф); полей магрезиста; петли *получения* (дроп/магазин); мутабельного лоадаута.

---

## 2. Несущие решения

1. **Гибрид:** авторские **уники** (нарративный «вау», под сюжетные точки) + простой **генерируемый пул** (массовый дроп и сток магазина).
2. **Эволюция — заменой.** Прокачка/крафт отложены (см. §8). Уники — главный аргумент за их возврат.
3. **Одна грамматика «джекпот ↔ пол» на весь лут** (гир и спеллы — см. §4–5): **дроп = джекпот** (вариативный, лучшее, бесплатно), **магазин = пол** (надёжная страховка за валюту).
4. **Лут бюджетируется дизайнером.** Нет оверворлда/рандом-боёв/фарма → тотал гира и валюты на главу авторски задан. «Рандом» — в том, *что* падает, не *сколько*. Баланс под контролем; ровно то, что хочет адаптивный директор.

---

## 3. Модель предмета

### Авторские уники
Рукотворные дефы (как сейчас). Несут «вау» и нарратив. Под чистую замену имеют
**срок годности** → пока прокачки нет, держим их актуальными:
- тюнингом на срок главы-двух, и/или
- **не-статовым свойством** (уникальный прок/способность), которое держит интерес даже когда обошли по числам.

### Генерируемый пул (держим маленьким)
База (существующие weapons/armor дефы) + **1–2 ролла** из небольшого аффикс-пула;
**рарити-тиры** гейтят число/силу аффиксов. Не Diablo-дерево.

**Аффикс-пул** (заземлён в существующем словаре + новое из прогрессии):
- сырые статы: СИЛ/ЛОВ/ТЕЛ/ИНТ, `armor`, `max_hp`, `spell_power`;
- **резисты:** 4 домена + общий (новые поля);
- **мастерство-выровненные:** `+spell_power домену`, `+эффективность стиля` и т.п. — *требует тегирования предмета доменом/стилем* (схемная работа).

**Тон:** под мрачноватый мир (bell_under_veil) генерируемое **приглушённое**, без
фейерверка легендарок. «Вау» несут уники, генерация — вариативность-наполнитель.

---

## 4. Гир: экономика

- **Дроп = джекпот:** лучшее/уникальное, вариативно, бесплатно. Источники: скриптовый дроп с энкаунтеров, сундуки/находки в сценах, награды за `Choice` (флаги уже есть).
- **Магазин = пол:** common/mid гир за валюту — закрыть невезучий слот, без него loot-variance × директор пересекаются зло.
- **Валюта** = награды за бой + **скупка ненужного гира** → петля: мёртвый лут (выпал для класса, которым не играю) → валюта → нужный лут.

---

## 5. Спеллы как лут

**Домен = гейт ёмкости** (какой тир домена ты вообще можешь освоить — сохраняет
ось ширины из `progression.md §3`). Свиток/обучение — *акт* выучивания.

Два источника, та же грамматика джекпот↔пол:

| Источник | Роль | Свойства |
|----------|------|----------|
| **Свиток (дроп)** | джекпот | бесплатно, оппортунистично, **может падать выше ёмкости** (аспирационно дорастаешь). Только падает — **не продаётся** (иначе сольётся с обучением) |
| **Обучение (магазин/гильдия)** | пол | стоит валюту, надёжно, **строго в пределах ёмкости** домена |

---

## 6. Магазин = сцен-хаб между главами

Не всегда-доступное меню, а **тип сцены** (городская/лагерная), привязанный к потоку
сценария — согласуется с no-grind структурой (зашёл, рестокнулся, пошёл дальше).

Функции хаба:
- гир-пол (купить),
- обучение спеллам (в пределах ёмкости домена),
- скупка ненужного (→ валюта).

Новый **тип сцены «хаб»** рядом со Story / Combat / Choice (`SceneDef`).

---

## 7. Менеджмент

Надеть/снять/продать/распределить по героям — в **roster/межбоевом экране**
(оправдан трижды: свободные очки прогрессии + менеджмент лута + доступ к хабу).
Партия именная → лут-решение = «кому отдать `+INT` посох». Держим просто:
надетые слоты на героя + общий сток, без inventory-тетриса.

---

## 8. Архитектурный шов (что держит лут)

Мутабельный per-character оверлей в сейве (**V2**, см. `progression.md §9`) дополнительно держит:
- **мутабельный лоадаут** (надетый гир поверх базы класса/шаблона),
- **сток/инвентарь** найденного,
- **выученные спеллы** (известные тома),
- **валюту** (на партию или на профиль).

Новое в данных/схеме:
- поля **магрезиста** (4 домена + общий) на `WeaponDef`/`ArmorDef`;
- **рарити/аффиксы** + генератор предметов;
- **тегирование доменом/стилем** для мастерство-выровненных аффиксов;
- **тип сцены «хаб»**.

## 9. Открытые вопросы
1. Числа: бюджет дропа/валюты по главе, число рарити-тиров, размер аффикс-пула.
2. Валюта — на партию или на профиль/слот?
3. Сток магазина — генерируется заново на каждый заход или фиксирован сценой?
4. Свиток выше ёмкости — насколько выше (на тир? без потолка?).

## 10. Отложено (отдельно)
- **Прокачка / крафт / рефордж** — долгожитие уников («меч Тео, что растёт с тобой»). Главный кандидат на следующий заход по луту.
- **Адаптивный директор** — балансирует дроп-бюджет и силу партии (общий с прогрессией пиллар).

## 11. MVP — минимальный первый срез

Цель среза: **дроп → инвентарь → смена предметов между боями.** Доказывает
архитектурный шов (`§8` / `progression.md §9`) — первый кусок мутабельного
оверлея в сейве. Всё «дорогое» отрезано (см. «вне среза» ниже).

### Что грунтовка показала (срез реально дешёвый)
- **`Equipment` компонент уже есть** (`main_hand/off_hand: Option<WeaponId>`, `chest/legs/feet: ArmorId`) и **уже питает статы** через `ContentView::effective_stats(base, equipment)` + `equipment_armor`. Поменял экипировку → статы пересчитались сами.
- **`spawn_combatants`** уже строит `Equipment` из класса/шаблона — нужна только ветка «если в сейве есть оверрайд, применить его».
- **Сейв** (`SaveSlotFile::V1` / `CampaignProgress`) уже использует `#[serde(default)]` для обратной совместимости → новые поля **аддитивны, bump до V2 не обязателен** (старые саве грузятся с пустым лоадаутом/стоком).

### Срез
1. **Дроп — авторский, детерминированный.** `EncounterDef` (или сцена) получает опц. `rewards: Vec<item_id>`. На Victory — предметы падают в сток. Без генерации/рарити/рандома (реюз существующих авторских дефов weapons/armor).
2. **Инвентарь — один плоский партийный сток.** `stash: Vec<ItemId>` в `CampaignProgress` (`#[serde(default)]`). Не персональные сумки.
3. **Лоадаут — оверлей на героя.** `loadouts: HashMap<character_id, EquipmentSave>` в `CampaignProgress` (`#[serde(default)]`). `character_id` = `PartyMemberDef.name` *(подтвердить стабильность/уникальность)*. На спавне: есть оверрайд → применить вместо дефолта класса.
4. **Смена между боями — минимальный экран экипировки.** Слоты героя ↔ сток; перенос с проверкой типа слота (оружие→рука, броня→свой слот) и hand-правил. Запись в `loadouts` (персист), эффект — со следующего спавна. Это **зерно лагеря** (`difficulty-recovery.md §3`) — остальные функции лагеря (хил/очки) сюда НЕ входят.

### Вне среза (отрезано осознанно)
Генерация/аффиксы/рарити · валюта/магазин/продажа · спеллы-как-лут/свитки/обучение ·
крафт/прокачка · лейн-ограничения гира (MVP: только тип слота) · полный лагерь.

### Прикидка по объёму
- Сейв-поля (2 аддитивных + структура `EquipmentSave`): **мелко**, паттерн есть.
- `rewards` + хук на Victory (append в сток): **мелко**.
- Оверрайд на спавне: **мелко** (ветка в `spawn_combatants`).
- **Экран экипировки: средне** — основной кусок, новый UI (перенос + валидация слота).
- Итого: **одна средняя UI-часть + несколько мелких data/runtime-хуков.** Срез
  де-рискует всю программу — последующее (оси, спеллы, валюта) расширяет тот же
  `loadouts`/оверлей-паттерн.

### Мелкие решения по срезу
- Где открывается экран экипировки: отдельный мини-экран между combat-сценами (рекомендую) / на Victory-экране / тоггл-панель?
- `character_id` = `PartyMemberDef.name` или нужен явный стабильный id?
