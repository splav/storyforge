# ТЗ на доработку движка для нужд ch2

Глава II «След медной книги» вводит четыре боя с новыми механиками,
которые не покрыты текущим движком (по состоянию ch1). Этот документ
описывает, что нужно добавить и в каком объёме.

Полный дизайн-док главы — `ch2/docs/description.md`.

---

## Сводка

| Механика | Бой | Приоритет | Сложность |
|---|---|---|---|
| Ловушки на местности | 1 | Высокий | Средняя |
| Двойная цель победы (kill_all + alive(NPC)) | 2 | Высокий | Низкая |
| NPC-объект на поле (статичный, без хода) | 2 | Высокий | Низкая |
| Преграды на гексах (блокируют стрельбу) | 3 | Высокий | Средняя |
| Дальнобойные атаки с проверкой преграды | 3 | Высокий | Низкая (поверх преград) |
| Смена цели победы между фазами | 4 | Средний | Низкая-Средняя |
| Поведение бегства у юнита | 4 | Средний | Средняя |

Всего: **7 пунктов**, из них критичных для геймплея — 5, остальные
можно упростить (см. секцию «Что можно упростить»).

---

## Что переиспользуется из ch1 (должно уже работать)

Проверено по конфигам глав I:

- VictoryTarget (`victory = { type = "kill_target", ... }`)
- Фазы юнитов (`[[encounters.enemies.phases]]` с `hp_below_pct`,
  `template`, `heal_to_full`, `flavor`)
- Аура (`aura = { status = "...", radius = N, affects = "enemies" }`)
- Призывы (`effect = "summon"`, `summon_template`, `summon_max_active`)
- Статусы (`disoriented`, `confuse`, и т.п.)
- `on_victory_flags`, `requires_flag` для условных диалогов
- `party_add` для динамического присоединения героев
- Атаки с дальностью (`range = N` у способностей)

ch2 опирается на этот фундамент. Ничего из этого менять не нужно.

## Новые механики

### 1. Ловушки на местности (бой 1: засада на тракте)

**Суть.** На конкретных гексах размещены ловушки, которые
срабатывают при входе **любого** юнита на клетку. Урон или
статус. Скрыты по умолчанию; раскрываются игроку через зону
обзора следопыта.

**Минимум для MVP.**

- Тип объекта «ловушка» с привязкой к гексу.
- Срабатывает на **любого** юнита (включая бандитов), кто наступит.
  Симметрия — единая логика входа на гекс.
- Одноразовый эффект (урон костями и/или статус), после
  срабатывания ловушка считается потраченной и **видна**.
- Видимость до срабатывания:
  - **Бандиты знают свои ловушки** — учитывают в AI (мягкий штраф
    в pathfinding, не запрет: если другого пути нет — пройдут).
  - **Партия видит ловушки в зоне обзора следопыта** (см. ниже).

**Зона обзора следопыта.**

- Юнит с флагом `reveals_traps_on_turn_start` (или тегом
  `ranger` — в нашей кампании это Kael).
- **В начале хода такого юнита** — запускается BFS на
  `movement_range` гексов от его текущей позиции. Это
  переиспользует существующий pathfinding для movement preview,
  никакого нового алгоритма.
- Все ловушки, попавшие в эту зону, помечаются `revealed = true`
  и остаются видимыми навсегда.
- Тактически: Kael играет как разведчик-вперёд. Его движение
  приносит информацию, не только урон. Партия может намеренно
  тратить его ход на разведку (выдвинуть вперёд, открыть зону,
  остальные пойдут безопасно). Если Kael мёртв — новые зоны не
  открываются.

**AI-сторона.**

- Pathfinding бандитов: гексы с известными им ловушками имеют
  мягкий штраф (например, +5 к стоимости). Если оптимальный путь
  всё равно ведёт через ловушку — пройдут (например, если другой
  путь длиннее на 10).
- Это позволяет партии **загонять врагов на ловушки** через
  тактическое давление (forced movement, перекрытие альтернатив).

**Что НЕ нужно.**

- Активные навыки сканирования и обезвреживания ловушек.
  Можно ввести позже, в ch3 или дальше.
- Разрушение ловушек (стрельбой, способностями).
- Ловушки с зональным эффектом (взрывы 2×2 и т.п.) — для ch2
  достаточно одной клетки.
- Каскадные ловушки (одна активирует другую).

**Примерный data-формат (для иллюстрации, не обязателен):**

```toml
[[encounters.environment]]
type = "trap"
id = "spike_trap_1"
hex_col = 4
hex_row = 3
hidden = true
trigger_on = "enter"
damage = { dice_count = 1, dice_sides = 6 }
# Опционально:
# statuses = [{ id = "bleeding", duration = 2 }]
# Стороны, которым ловушка известна на старте боя (для AI):
known_to_sides = ["enemy"]
```

В шаблоне юнита-следопыта:

```toml
# В unit_templates.toml для класса ranger:
reveals_traps_on_turn_start = true
```

### 2. Двойная цель победы + NPC-объект (бой 2: тронутое святилище)

**Суть.** Условие победы — убить всех врагов **И** обеспечить, что
конкретный NPC остался живым к концу боя. NPC сам не действует:
лежит на гексе без сознания, без хода, может быть убит.

Это две отдельные подзадачи, которые удобно делать вместе.

#### 2.1 NPC-объект на поле

**Минимум.**

- Тип юнита «лежащий NPC»: имеет HP, может быть целью атаки,
  но **не имеет хода** (не активируется в очереди инициативы).
- Привязан к гексу, не двигается.
- Принадлежит «партии» или «нейтралам», но **враги могут его
  атаковать** (как нейтральную или партийную цель).
- Имеет имя/template, чтобы можно было ссылаться в условиях
  победы.

**Примерный data-формат:**

```toml
[[encounters.npcs]]
template = "wounded_magister"
hex_col = 6
hex_row = 4
hp_current = 6
hp_max = 6
side = "ally"
acts = false  # не участвует в очереди ходов
```

#### 2.2 Двойная цель победы

**Минимум.**

- Расширение `victory` так, чтобы поддерживать **И**-комбинацию
  условий.
- Достаточно поддержать два типа в комбинации: `kill_all`
  (вся вражеская сторона мертва) и `alive` (указанный юнит/NPC
  жив).

**Примерный data-формат:**

```toml
victory = { type = "all_of", conditions = [
  { type = "kill_all" },
  { type = "alive", target = "wounded_magister" },
] }
```

Поражение боя — если выполнено любое условие, не выполнимое
дальше (например, NPC умер раньше окончания боя).

### 3. Преграды на гексах + дальнобойные атаки с их учётом
(бой 3: портовый квартал)

**Суть.** В городской застройке дальнобойные враги стреляют через
карту, но не сквозь препятствия. Партия учится использовать ящики,
повозки, углы домов для прикрытия.

#### 3.1 Преграды на гексах

**Минимум.**

- Тип объекта «преграда» с привязкой к гексу.
- Преграда **блокирует движение** через гекс (нельзя пройти).
- Преграда **блокирует дальнобойные атаки** через гекс
  (стрелять «сквозь» нельзя).
- Не имеет HP, не разрушается — статичный декор поля.

**Примерный data-формат:**

```toml
[[encounters.environment]]
type = "obstacle"
hex_col = 5
hex_row = 3
```

Сразу несколько в одном бою.

#### 3.2 Атаки с проверкой преграды

**Минимум (упрощённый LOS).**

При дальнобойной атаке (range > 1) — провести **прямую линию
по гексам** от атакующего до цели и проверить, что **ни на одном
промежуточном гексе нет преграды**. Если есть — атака недоступна
(в UI — невозможно выбрать цель).

Достаточно простой алгоритм без подсчёта углов:
интерполяция гексов на линии между двумя точками. В литературе
по гексагональным сеткам этот алгоритм называется «hex line drawing»;
имплементация — десятки строк.

**Что НЕ нужно.**

- Полноценная LOS с подсчётом по углам, частичными перекрытиями,
  высотой.
- Низкие/высокие препятствия (всё блокирует одинаково).
- Тенью прикрытие, частичное укрытие, бонусы к защите за укрытием.

Это упрощение — преграда либо есть, либо нет. Партия учится
выбирать позицию, чтобы между ней и стрелком был хотя бы один
гекс-препятствие.

### 4. Смена цели победы между фазами + поведение бегства
(бой 4: подземные склады)

**Суть.** Босс (Колм) при определённом условии переходит во вторую
фазу: цель победы меняется на «убить Колма за N ходов», сам Колм
перестаёт драться и каждый ход отходит от партии.

#### 4.1 Смена цели победы по триггеру

**Минимум.**

- Расширение `[[encounters.enemies.phases]]` так, чтобы фаза могла
  не только менять template/HP юнита, но и **переопределять
  условие победы боя**.
- Триггер фазы — как сейчас, `hp_below_pct`.

**Примерный data-формат:**

```toml
[[encounters.enemies.phases]]
hp_below_pct = 30
flavor = "Колм бросает оружие и пытается уйти."
# Меняем поведение юнита:
ai_behavior = "flee"
# Меняем условие победы боя:
victory_override = { type = "kill_target", enemy_name = "Тибор Колм", turn_limit = 4 }
```

После N ходов от старта фазы 2 — если Колм жив, считается, что
сбежал (новое условие провала боя).

#### 4.2 Поведение «бегство»

**Минимум.**

Новый AI-поведенческий режим: юнит каждый ход выбирает действие
«движение в максимальное удаление от ближайшего врага» (игнорирует
атаку). При достижении края карты — стоит на месте (или продолжает
жаться к краю).

Не нужна сложная карта с «выходом» — достаточно «уйти от партии
максимально далеко». Через N ходов считается сбежавшим.

**Что НЕ нужно.**

- Прокладка маршрута к конкретной точке выхода.
- Сложные обходы препятствий (если попал в угол — стоит).
- Реакция бегущего юнита на атаки (он не отвечает).

## Приоритеты

### Критичные (без них ch2 не играется)

1. **Двойная цель победы + NPC-объект** (бой 2). Без этого бой 2
   не имеет смысла — некого спасать.
2. **Преграды на гексах + LOS** (бой 3). Без этого бой 3 — просто
   ещё одна свалка, теряет тематику.
3. **Смена цели победы между фазами** (бой 4). Без этого финал
   главы — обычный бой с боссом, теряет драматику.

### Желательные (упрощают подачу, но можно обойти)

4. **Ловушки на местности** (бой 1). Если не успеваем — бой 1
   можно сделать как стандартный «убей всех», флавор-функция Kael'а
   подаётся через сюжетный текст до боя.
5. **Поведение «бегство»** (бой 4). Если не успеваем — Колм во
   фазе 2 может просто стоять на месте без атак, и партия должна
   успеть его убить за N ходов. Структурно работает, теряется
   только визуальная динамика бегства.

### Можно отложить полностью

6. **Видимость ловушек по тегу класса** (бой 1). Подаётся через
   нарратив, не через механику.

## Что осознанно НЕ требуется для ch2

Чтобы избежать раздувания скоупа:

- **Активный escort** (NPC двигается, враги активно идут его
  убить). У нас NPC статичен и пассивен — это сильно проще.
- **Активируемые объекты на карте** (рычаги, бомбы со Слёзами,
  меняющиеся проходы). Изначально рассматривали — отказались.
- **Сложная LOS** с углами, частичными укрытиями, высотами.
  У нас бинарное: преграда есть/нет.
- **Разрушаемое окружение** (стрелять по ящикам, чтобы убрать
  укрытие). Преграды статичны.
- **Несколько одновременных VictoryTarget** в боже одной фазы.
  Используется максимум один VictoryTarget на фазу.
- **Поведение бегства с прокладкой пути к выходу.** Достаточно
  «уйти максимально далеко».
- **Реактивные триггеры в диалогах** (диалог-битва как в ch3
  бой 3). Это уже для ch3.

## Замечания по структуре конфигов

Текущие конфиги ch1 хорошо ложатся в новые механики — большинство
расширений идут как дополнительные поля или новые секции,
не ломающие совместимость.

Предложенный формат с `[[encounters.environment]]` (преграды,
ловушки) — пример, разработчик может выбрать другую структуру
(например, отдельный файл `environment.toml` на сценарий).
Главное — данные про окружение должны быть **сценарно-локальны**,
как сейчас локальны враги и юниты.

Если будут вопросы по конкретным сценам — см. полный дизайн-док
`ch2/docs/description.md`.

## Архитектурные решения (зафиксировано 2026-05-27)

Финал по развилкам после планирования. Имплементер опирается на этот
раздел, не переоткрывая выборы.

### Скоуп и оценка

| Волна | Содержание | Оценка |
|---|---|---|
| 1 | NPC-объект + двойная victory ║ преграды + LOS | 4–5д |
| 2 | Phase override + turn_limit ║ flee AI через `EvaluationMode::Flee` | 4–5д |
| 3 | Env-подсистема (hazards + knowledge + Kael reveal + severity pathfinding) | 5–7д |
|   | **Итого** | **13–17 чел-дней** |

Волна 3 — переиспользуемая инвестиция: в ch3 на тот же примитив
ложатся биоразливы и зона осколка Слезы.

### 1. Env-объекты и опасные гексы (волна 3)

Универсальный примитив, **не ad-hoc для ловушек**.

- **Сущность.** `EnvObject { id, kind: Hazard|Obstacle, hex, triggered: bool, revealed: bool, effect, severity_override: Option<f32> }`.
- **Хранение.** В pure engine, `CombatState.environment: Vec<EnvObject>`. Schema bump до v43.
- **Источник эффектов ловушки.** Не phantom-unit. Новый вариант
  `DamageSource::Env(EnvId)` / `StatusApplier::Env(EnvId)`. Это требует
  правок в rage-rules (от env урона ярость **не копится** ни у кого) и
  в AoO-консьюмерах. Сделать в одном PR со schema bump.
- **События.** `Event::HazardTriggered { env_id, victim }`, `Event::EnvRevealed { env_id }`.
- **Триггер.** В `apply_effect(MovePosition)` после установки позиции —
  скан `environment`: если на новой клетке есть `kind=Hazard, triggered=false`
  и юнит уязвим — применяется effect, `triggered=true`.
- **Контент.** Секция `[[encounters.environment]]` в TOML энкаунтера.

### 2. Knowledge model (волна 3)

Per-unit модель «кто знает про какие hazards».

- **Поле на юните.** `Unit.knows_env: KnowledgeMode ∈ { AllInEncounter, RevealedOnly }`.
- **Дефолт.** Партия и нейтралы — `RevealedOnly`. Враги-владельцы (бандиты
  в их собственной засаде) — `AllInEncounter`. Дикие/одержимые —
  `RevealedOnly`.
- **Контент.** Поле `knows_env = "all"|"revealed_only"` на шаблоне юнита
  или override-ом в энкаунтере.

### 3. Kael perception (волна 3)

Пассив на классе/теге. Не active ability.

- **Маркер.** Тег юнита `reveals_traps_on_turn_start` (или флаг на
  классе ranger/scout).
- **Логика.** На `TurnStarted` для такого юнита: запустить тот же BFS,
  что используется в movement preview, на `movement_range` гексов от
  текущей позиции. Все hazards с координатами внутри результата —
  `revealed = true` (permanent).
- **Видимость.** Party-wide. После reveal hazard рендерится в UI всем
  и всем считается известным для pathfinding (party-side knowledge).
- **Структурная цена.** Если Kael мёртв — новые зоны не открываются.
  Эта зависимость намеренная: разведчик ценен живым.
- **Переиспользование.** Никакого нового алгоритма — тот же
  `MovementEnv` + BFS из `pathfinding.rs`.

### 4. Severity для AI pathfinding (волна 3)

**Severity ловушки — derived, не вручную.**

- **Формула (baseline).**
  ```
  severity = expected_damage_avg + Σ status_cost(status_i, duration_i)
  ```
  где:
  - `expected_damage_avg = dice_count * (dice_sides + 1) / 2` (для урона ловушки);
  - `status_cost(...)` — переиспользует уже существующую AI-оценку статусов
    (см. `combat/ai/scoring/` — там есть базовый weight per status,
    использовать его как источник истины).
- **Override.** Опциональное поле `severity_override: f32` в TOML —
  на случай если контент-дизайнеру нужна нестандартная оценка
  (фикспоинт-механика «капкан страшнее по нарративу чем по урону»).
- **Применение в pathfinding.**
  `MovementEnv` получает `hazard_costs: HashMap<Hex, f32>` —
  построенный bridge-side из `environment` с фильтром по `knows_env`
  юнита-планировщика. A* умножает базовый cost-of-step на
  `1 + severity_multiplier(hazard_cost)`. Если альтернативы нет — идёт.
- **Что НЕ нужно.** Не считать severity внутри pure engine — это
  AI-концепция. Engine хранит только raw effect ловушки. Severity
  вычисляется bridge-side при формировании `MovementEnv` для AI.

### 5. NPC-объект (волна 1)

- **Сторона.** `Team::Player` (не вводим `Team::Neutral`).
- **Маркер.** Новый компонент `NonActingNpc` на entity.
- **Очередь хода.** `build_turn_order` фильтрует юнитов с этим маркером.
- **Лечение.** NPC лечится партией штатно (AI-эвристики лечат как ally).
  Магистр в бою 2 = неподвижный союзник без действий, **с возможностью
  быть исцелённым**.
- **`players_alive` подсчёт.** В `check_combat_end` исключить
  `NonActingNpc` из подсчёта «партия жива» — иначе можно проиграть
  бой, оставшись только с лежащим NPC.

### 6. Двойная victory (волна 1)

- **Enum.**
  ```rust
  enum VictoryCondition {
      AllEnemiesDead,
      KillTarget { ... },
      KeepAlive { target_name: String, marker_color: [f32; 3] },
      AllOf(Vec<VictoryCondition>),
  }
  ```
  `AllOf` оценивается рекурсивно в `determine_outcome`.
- **`KeepAlive` семантика.** Юнит мёртв → бой провален немедленно
  (не дожидаемся остальных условий). Это leaf-condition, не conjunction-member.
- **UI HUD.** `objective_text()` обходит дерево, выводит компоненты через «И».

### 7. Преграды + LOS (волна 1)

- **Engine знает про preграды для движения.** `CombatState.blocked_hexes: HashSet<Hex>`.
  Schema bump (объединить с env schema bump если волны идут параллельно).
- **LOS-функция.** Использовать **существующую** `has_los` из
  `src/game/hex.rs` (она готова и покрыта тестами).
- **ActionState трейт.** Добавить метод `is_blocked_los(from, to) -> bool`
  с default-impl `false`. Реализации `BevyActions` и `SnapshotActions`
  **вызывают одну и ту же функцию `has_los`** — гарантия parity между
  prod-сценой и AI sim. Покрыть property-тестом на parity.
- **AbilityDef.** Поле `requires_los: bool` (default false). Атаки с
  `range > 1` чаще всего получат `true`.
- **IllegalReason.** Новый вариант `NoLineOfSight`.

### 8. Phase victory override + turn_limit (волна 2)

- **Counter.** **Round-based.** Не actor-turn-based.
- **Хранение.** `PhaseDef.victory_override: Option<VictoryCondition>`,
  `PhaseDef.turn_limit: Option<u32>`.
- **Активация.** `apply_phase_ecs_writes` при срабатывании фазы
  переписывает ресурс `CombatObjective` и устанавливает ресурс
  `PhaseDeadline { phase_started_round: u32, limit: u32 }`.
- **Проверка.** Новая система `check_phase_deadline_system` в Finalize
  set (или встроить в `check_victory_system`). Условие истечения:
  `current_round - phase_started_round >= limit` И target жив → defeat
  (босс сбежал).
- **Engine не трогаем.** `PhaseEntry` в engine остаётся as-is
  (статы/heal). Логика смены цели — bridge-side.

### 9. Flee AI (волна 2)

- **Реализация через pipeline, не bypass.** Новый `EvaluationMode::Flee`.
  Обоснование: в ch3+ будут другие AI-режимы (паника, патруль и т.п.),
  лучше иметь правильный путь сразу.
- **Активация.** Поле `PhaseDef.ai_behavior: Option<String>` (значение
  `"flee"` для фазы 2 Колма). `apply_phase_ecs_writes` навешивает на
  entity компонент `FleeingBehavior`.
- **Pipeline.** `mode_selection` стадия проверяет компонент перед
  обычной axis-evaluation. В режиме `Flee`:
  - factor `flee_distance` с очень большим положительным весом
    (максимизировать дистанцию до ближайшего врага);
  - mask на все offensive abilities (Cast-кандидаты выкидываются);
  - heal/buff на себя — разрешены (босс может пытаться лечиться при
    бегстве).
- **Edge cases.** В углу карты (нет хода дальше) — EndTurn без Move,
  не зависнуть.
- **Sim consistency.** AI партии моделирует противника-в-режиме-Flee
  тем же `EvaluationMode::Flee` через `apply_step`, что и engine
  использует для исполнения — режим виден симуляции.

### 10. Закрытые развилки (не делаем)

- Phantom-unit для env-объектов — отклонено в пользу прямой engine-сущности
  `EnvObject` с `DamageSource::Env`.
- `Team::Neutral` для NPC — отклонено, используем `Team::Player + NonActingNpc`.
- Actor-turn-based `turn_limit` — отклонено, round-based.
- Bypass для flee — отклонено, идём через `EvaluationMode::Flee`.
- Видимость ловушек по тегу класса как отдельная фича — заменена на
  «Kael BFS-reveal на старте хода», это богаче и переиспользует
  существующий pathfinding.
- Severity ловушек вручную в TOML — derived из урона+статуса,
  override только опционально.
