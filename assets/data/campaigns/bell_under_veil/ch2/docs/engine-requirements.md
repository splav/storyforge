# ТЗ на доработку движка для нужд ch2

Глава II «След медной книги» вводит четыре боя с новыми механиками,
которые не покрыты текущим движком (по состоянию ch1). Этот документ
описывает, что нужно добавить и в каком объёме.

Полный дизайн-док главы — `ch2/docs/description.md`.

---

## Сводка

| Механика | Бой | Приоритет | Сложность |
|---|---|---|---|
| Ловушки на местности | 1 | Высокий | Средняя |
| Двойная цель победы (kill_all + alive(NPC)) | 2 | Высокий | Низкая |
| NPC-объект на поле (статичный, без хода) | 2 | Высокий | Низкая |
| Преграды на гексах (блокируют стрельбу) | 3 | Высокий | Средняя |
| Дальнобойные атаки с проверкой преграды | 3 | Высокий | Низкая (поверх преград) |
| Смена цели победы между фазами | 4 | Средний | Низкая-Средняя |
| Поведение бегства у юнита | 4 | Средний | Средняя |

Всего: **7 пунктов**, из них критичных для геймплея — 5, остальные
можно упростить (см. секцию «Что можно упростить»).

## Что переиспользуется из ch1 (должно уже работать)

Проверено по конфигам глав I:

- VictoryTarget (`victory = { type = "kill_target", ... }`)
- Фазы юнитов (`[[encounters.enemies.phases]]` с `hp_below_pct`,
  `template`, `heal_to_full`, `flavor`)
- Аура (`aura = { status = "...", radius = N, affects = "enemies" }`)
- Призывы (`effect = "summon"`, `summon_template`, `summon_max_active`)
- Статусы (`disoriented`, `confuse`, и т.п.)
- `on_victory_flags`, `requires_flag` для условных диалогов
- `party_add` для динамического присоединения героев
- Атаки с дальностью (`range = N` у способностей)

ch2 опирается на этот фундамент. Ничего из этого менять не нужно.

## Новые механики

### 1. Ловушки на местности (бой 1: засада на тракте)

**Суть.** На конкретных гексах размещены ловушки, которые
срабатывают при входе юнита на клетку. Урон или статус.
По умолчанию **скрыты** — игрок их не видит до срабатывания
или до явного обнаружения.

**Минимум для MVP.**

- Тип объекта «ловушка» с привязкой к гексу.
- При входе **любого юнита** (включая своих) на гекс — срабатывание:
  одноразовый эффект (урон бросками костей и/или статус), после
  чего ловушка считается потраченной.
- Видимость ловушки до срабатывания — по умолчанию скрыта
  (не рендерится в UI), после срабатывания — видна.

**Желательно (опционально).**

- Флаг `visible_to_tag = "ranger"` или аналогичный механизм,
  позволяющий конкретному классу/тегу видеть ловушки заранее
  (флавор-функция Kael'а из лора). Если сложно — можно отложить
  и реализовать через сюжетный bit «следопыт показывает одну
  ловушку перед началом боя» как сцену, без механики.

**Примерный data-формат (для иллюстрации, не обязателен):**

```toml
[[encounters.environment]]
type = "trap"
id = "spike_trap_1"
hex_col = 4
hex_row = 3
hidden = true
trigger_on = "enter"
damage = { dice_count = 1, dice_sides = 6 }
# Опционально:
# statuses = [{ id = "bleeding", duration = 2 }]
# visible_to_tag = "ranger"
```

### 2. Двойная цель победы + NPC-объект (бой 2: тронутое святилище)

**Суть.** Условие победы — убить всех врагов **И** обеспечить, что
конкретный NPC остался живым к концу боя. NPC сам не действует:
лежит на гексе без сознания, без хода, может быть убит.

Это две отдельные подзадачи, которые удобно делать вместе.

#### 2.1 NPC-объект на поле

**Минимум.**

- Тип юнита «лежащий NPC»: имеет HP, может быть целью атаки,
  но **не имеет хода** (не активируется в очереди инициативы).
- Привязан к гексу, не двигается.
- Принадлежит «партии» или «нейтралам», но **враги могут его
  атаковать** (как нейтральную или партийную цель).
- Имеет имя/template, чтобы можно было ссылаться в условиях
  победы.

**Примерный data-формат:**

```toml
[[encounters.npcs]]
template = "wounded_magister"
hex_col = 6
hex_row = 4
hp_current = 6
hp_max = 6
side = "ally"
acts = false  # не участвует в очереди ходов
```

#### 2.2 Двойная цель победы

**Минимум.**

- Расширение `victory` так, чтобы поддерживать **И**-комбинацию
  условий.
- Достаточно поддержать два типа в комбинации: `kill_all`
  (вся вражеская сторона мертва) и `alive` (указанный юнит/NPC
  жив).

**Примерный data-формат:**

```toml
victory = { type = "all_of", conditions = [
  { type = "kill_all" },
  { type = "alive", target = "wounded_magister" },
] }
```

Поражение боя — если выполнено любое условие, не выполнимое
дальше (например, NPC умер раньше окончания боя).

### 3. Преграды на гексах + дальнобойные атаки с их учётом
(бой 3: портовый квартал)

**Суть.** В городской застройке дальнобойные враги стреляют через
карту, но не сквозь препятствия. Партия учится использовать ящики,
повозки, углы домов для прикрытия.

#### 3.1 Преграды на гексах

**Минимум.**

- Тип объекта «преграда» с привязкой к гексу.
- Преграда **блокирует движение** через гекс (нельзя пройти).
- Преграда **блокирует дальнобойные атаки** через гекс
  (стрелять «сквозь» нельзя).
- Не имеет HP, не разрушается — статичный декор поля.

**Примерный data-формат:**

```toml
[[encounters.environment]]
type = "obstacle"
hex_col = 5
hex_row = 3
```

Сразу несколько в одном бою.

#### 3.2 Атаки с проверкой преграды

**Минимум (упрощённый LOS).**

При дальнобойной атаке (range > 1) — провести **прямую линию
по гексам** от атакующего до цели и проверить, что **ни на одном
промежуточном гексе нет преграды**. Если есть — атака недоступна
(в UI — невозможно выбрать цель).

Достаточно простой алгоритм без подсчёта углов:
интерполяция гексов на линии между двумя точками. В литературе
по гексагональным сеткам этот алгоритм называется «hex line drawing»;
имплементация — десятки строк.

**Что НЕ нужно.**

- Полноценная LOS с подсчётом по углам, частичными перекрытиями,
  высотой.
- Низкие/высокие препятствия (всё блокирует одинаково).
- Тенью прикрытие, частичное укрытие, бонусы к защите за укрытием.

Это упрощение — преграда либо есть, либо нет. Партия учится
выбирать позицию, чтобы между ней и стрелком был хотя бы один
гекс-препятствие.

### 4. Смена цели победы между фазами + поведение бегства
(бой 4: подземные склады)

**Суть.** Босс (Колм) при определённом условии переходит во вторую
фазу: цель победы меняется на «убить Колма за N ходов», сам Колм
перестаёт драться и каждый ход отходит от партии.

#### 4.1 Смена цели победы по триггеру

**Минимум.**

- Расширение `[[encounters.enemies.phases]]` так, чтобы фаза могла
  не только менять template/HP юнита, но и **переопределять
  условие победы боя**.
- Триггер фазы — как сейчас, `hp_below_pct`.

**Примерный data-формат:**

```toml
[[encounters.enemies.phases]]
hp_below_pct = 30
flavor = "Колм бросает оружие и пытается уйти."
# Меняем поведение юнита:
ai_behavior = "flee"
# Меняем условие победы боя:
victory_override = { type = "kill_target", enemy_name = "Тибор Колм", turn_limit = 4 }
```

После N ходов от старта фазы 2 — если Колм жив, считается, что
сбежал (новое условие провала боя).

#### 4.2 Поведение «бегство»

**Минимум.**

Новый AI-поведенческий режим: юнит каждый ход выбирает действие
«движение в максимальное удаление от ближайшего врага» (игнорирует
атаку). При достижении края карты — стоит на месте (или продолжает
жаться к краю).

Не нужна сложная карта с «выходом» — достаточно «уйти от партии
максимально далеко». Через N ходов считается сбежавшим.

**Что НЕ нужно.**

- Прокладка маршрута к конкретной точке выхода.
- Сложные обходы препятствий (если попал в угол — стоит).
- Реакция бегущего юнита на атаки (он не отвечает).

## Приоритеты

### Критичные (без них ch2 не играется)

1. **Двойная цель победы + NPC-объект** (бой 2). Без этого бой 2
   не имеет смысла — некого спасать.
2. **Преграды на гексах + LOS** (бой 3). Без этого бой 3 — просто
   ещё одна свалка, теряет тематику.
3. **Смена цели победы между фазами** (бой 4). Без этого финал
   главы — обычный бой с боссом, теряет драматику.

### Желательные (упрощают подачу, но можно обойти)

4. **Ловушки на местности** (бой 1). Если не успеваем — бой 1
   можно сделать как стандартный «убей всех», флавор-функция Kael'а
   подаётся через сюжетный текст до боя.
5. **Поведение «бегство»** (бой 4). Если не успеваем — Колм во
   фазе 2 может просто стоять на месте без атак, и партия должна
   успеть его убить за N ходов. Структурно работает, теряется
   только визуальная динамика бегства.

### Можно отложить полностью

6. **Видимость ловушек по тегу класса** (бой 1). Подаётся через
   нарратив, не через механику.

## Что осознанно НЕ требуется для ch2

Чтобы избежать раздувания скоупа:

- **Активный escort** (NPC двигается, враги активно идут его
  убить). У нас NPC статичен и пассивен — это сильно проще.
- **Активируемые объекты на карте** (рычаги, бомбы со Слёзами,
  меняющиеся проходы). Изначально рассматривали — отказались.
- **Сложная LOS** с углами, частичными укрытиями, высотами.
  У нас бинарное: преграда есть/нет.
- **Разрушаемое окружение** (стрелять по ящикам, чтобы убрать
  укрытие). Преграды статичны.
- **Несколько одновременных VictoryTarget** в боже одной фазы.
  Используется максимум один VictoryTarget на фазу.
- **Поведение бегства с прокладкой пути к выходу.** Достаточно
  «уйти максимально далеко».
- **Реактивные триггеры в диалогах** (диалог-битва как в ch3
  бой 3). Это уже для ch3.

## Замечания по структуре конфигов

Текущие конфиги ch1 хорошо ложатся в новые механики — большинство
расширений идут как дополнительные поля или новые секции,
не ломающие совместимость.

Предложенный формат с `[[encounters.environment]]` (преграды,
ловушки) — пример, разработчик может выбрать другую структуру
(например, отдельный файл `environment.toml` на сценарий).
Главное — данные про окружение должны быть **сценарно-локальны**,
как сейчас локальны враги и юниты.

Если будут вопросы по конкретным сценам — см. полный дизайн-док
`ch2/docs/description.md`.
