## Б.1 — Persistent-исходы боёв и вторичные цели + сценарный проигрыш ✅

### Проблема (как сейчас)

Флаги **выводимые**, а не хранимые: `active_flags(scen, scene_index)` сканирует
сцены до текущей и собирает `on_victory_flags` пройденных боёв, предполагая
линейную победу. `CampaignState` = `{ campaign_id, scenario_index }` — хранилища
достижений нет. Следствия:
- флаги не переживают границу главы (эпилог ch3 не видит, что было раньше);
- флаг **безусловен** — нельзя выразить «лодка выжила / погибла»;
- проигрыш всегда = рестарт боя; **нельзя проиграть и пойти дальше по сюжету**.

### Решение

Поднять флаговый механизм до **campaign-persistent** (хранится в сейве) и свести
авторскую модель к двум сущностям на бой:

1. **Исход боя** — одно значение, пишется движком автоматически на конце боя:
   `<scenario>.<bout>.result = victory | defeat`. Авторить не нужно.
2. **Достигнутые вторичные цели** — список id. Авторятся условиями, на конце
   боя прогоняются, достигнутые пишутся в набор.

Это **убирает** прежние `on_victory_flags` / `on_defeat_flags`:
- маркеры победы → растворяются в `result == victory`;
- «нашли жетон / спасли X» → переезжают во вторичные цели с тем же id.

#### Изменения

**`CampaignState`** (сериализуется в сейв):
- `results: Map<BoutKey, Outcome>` — `Outcome = Victory | Defeat`.
- `achieved: Set<ObjectiveId>` — достигнутые вторичные цели.
- (Под-развилка: возможно свести оба в единый `flags: Set<String>`; решить при реализации.)

**Combat-сцена (авторские поля):**
- `on_defeat = "retry" | "proceed"` — **дефолт `retry`** (всё текущее поведение не трогаем).
  `proceed` → поражение ведёт к следующей сцене.
- `[[encounters.objectives]] { id, condition, hidden }` — `condition` переиспользует
  leaf-ы `VictoryCondition` (напр. `KeepAlive`-чек «объект жив в конце»);
  оценивается на конце боя по финальному стейту.

**Поведение на конце боя (engine/bridge):**
1. записать `result`;
2. прогнать `objectives` по финальному стейту → записать достигнутые id
   (оценивать при **любом завершённом** исходе: победа ИЛИ proceed-поражение);
3. при `defeat` + `proceed` → продвинуть сценарий **и сбросить боевые статы
   партии к шаблону** перед следующим боем (сюжетные последствия — текстом + флагами);
   при `defeat` + `retry` → рестарт боя (как сейчас).

**Чтение:** `requires_flag` и эпилог-развилки читают из persistent-набора —
id целей напрямую, маркеры победы через `result == victory`.

#### Миграция

Заменить выводимый `active_flags` на чтение из хранилища. Три читаемых флага ch1
(`found_glassworks_token`, `kael_found`, `novice_saved`) → становятся id вторичных
целей с теми же строками → `requires_flag` не ломается. Неиспользуемые маркеры
(`bridge_routed`, `*_cleared`, …) — никто не читает, переводятся в `result` или дропаются.

#### Применение в ch3 — конкретный список флагов/целей

| Флаг / цель | Где ставится | Значения | Читает |
|---|---|---|---|
| `boat_saved` | бой 1, objective (лодка жива на победе) | да / нет | альт-эпилог (партия не уходит штатно) |
| `theo_fate` | бой 3, исход | `killed` / `surrendered` | эпилог, арка Лиры |
| `marken_fate` | бой 3, выбор (бить линию подопытных vs прорыв к Тэо) | `killed` / `possessed` | эпилог, арка Каэла |
| `kasian_choice` | эпилог, выбор | `accept` / `refuse` / `strike` | финал кампании; хук для продолжения |

- **Лодка:** `NonActingNpc` с HP; её гибель **≠ проигрыш** — пишет неудачу цели `boat_saved`.
- **Тэо** не входит в бой 4 ни при каком исходе (ростер Б.4 фиксирован).
- **Маркен** — без ветки «в партию»; только флаг → текст эпилога. По лору чистого
  спасения нет (даже «спасённый» — с Гостем внутри, см. `theo.md`).

#### Открытые под-вопросы

- Точная форма хранилища (единый `flags` vs `results` + `achieved`).
- Схема `BoutKey` (неймспейс-строки `scenario.bout`).
- Нужны ли latched-цели «событие случилось по ходу боя» (пока — только end-state чек).

## Б.2 — Пастух / симбионт (heal-юниты, бой 2) ✅

Цель сцены: научить «бей лекаря в тылу, а не танка, который не падает».

### Решённая модель

Лечение — **прямой каст обычного юнита, выбираемый AI** (не аура, не пассив,
не статус-реген, не переливание маны — все эти варианты рассмотрены и отклонены
как более дорогие и/или усложняющие AI; см. историю дизайна).

**Симбионт:**
- тег **`symbiote`** (нужен для предиката цели хила);
- броня **3** (`heavy_plate`-эквивалент), HP **~16**, средний урон, **без самоотхила**.

**Пастух:**
- **1 AP**, броня **0-1**, HP **~10-12**, mana **~8-10**, низкий урон;
- способности:
  - ближняя атака (`weapon_attack`);
  - дальняя магатака — клон `spark`/`flash`, 1 мана, слабее вспышки (ситуативно);
  - **хил симбионта:** `single_ally`, `cost_ap=1`, 1 мана, эффект `heal` **+3 HP**,
    **только цель с тегом `symbiote`**.

При 1 AP пастух делает одно действие/ход — почти всегда хил; атаки включаются,
когда лечить некого. Потому большой пул маны не нужен (тратит ~1/ход).

### Движок — tag-предикаты (Атом 3, общий с Б.4)

**`requires_tags` на способности** + проверка в `legality`. Хил пастуха: `requires_tags = {symbiote}`.
Это часть общего tag-атома (`Unit.tags: Set<Tag>` + предикаты + add/remove-tag; полная
спека — `engine-tz.md` Атом 3). Остальное (heal-эффект, `costs`, `cost_ap`, `single_ally`,
пулы, броня) — **существующий контент**.

- **AI наследует предикат бесплатно** (берёт цели через `legality`) — подтвердить при реализации.

### Баланс (проверен на числах)

Формула брони — плоское вычитание: `actual = max(1, raw − armor + dmg_taken)`,
магия (pierces) броню игнорирует. У всех юнитов **1 AP/ход** по умолчанию.

| Боец | Атака (vs броня 3) |
|---|---|
| Aldric | long_sword 2d6+4 ≈ **8** (физ) |
| Kael | bow 1d8+1 ≈ **2.5** (физ) + яд ~2.5/тик (pierce) |
| Lyra | spark 1d3+3+1 = **6** (pierce) |

- **Фокус втроём по одному:** ~16.5/ход ≫ хил → симбионт (HP 16) падает за ~1.3 хода.
  Хил **не делает симбионтов неубиваемыми** — верно.
- **Размаз (по 1 атакующему):** хил **+3** полностью гасит Каэлов выстрел (2.5) →
  такой симбионт фактически бессмертен → **урок «не размазывай, сними лекаря или
  фокусь»** возникает органично. Лира (pierce) — структурный ответ, её урон хил не отменяет.
- **+3 — потолок** (при +4 буксует даже фокус). Пастух 1 AP сознательно выбран,
  чтобы избежать feel-bad «воин бьёт в пустоту» (2 AP = +6/ход почти нейтрализует
  одиночного Aldric).

> Числа выше — под **троих**. В бою 2 партия уже **вчетвером** (с Ореном) и Aldric
> ослаблен (`injured`); Орен урона почти не даёт → дамагеров ~3. Итоговый ребаланс —
> см. **Б.5**. Симметрия «свой лекарь vs их пастух» урок не ломает.

### Verify-задача (часть Б.2)

Подтвердить, что вражеский AI **вменяемо выбирает heal** на раненого союзника
(ability + скоринг уже есть). Если плохо — чинить скоринг; пассив-фолбэк держим
в уме, но он дороже.

## Б.3 — Переговоры / бой с Тэо (бой 3) ✅

**Ключевой сдвиг:** НЕ строим in-combat «диалог-битву». Тэо предлагает переговоры
**до** боя (он спокоен, ждёт их) → выбор естественно **предбоевой**. Механика
раскладывается на дешёвое:

1. **Story-выбор игрока → флаг — единственный новый примитив.** Story-слой сейчас
   линейный (`Story{lines}` / `Combat`, без ветвлений). Нужен примитив: сцена
   с вариантами → ветвление по флагу.
   **Нужен эпилогу в любом случае** (`kasian_choice` accept/refuse/strike — тот же
   примитив) → строим один раз, закрываем Б.3 И финал кампании.
2. **Ветка «переговоры»** → Тэо сдаётся в story-сцене, бой пропускается,
   `theo_fate=surrendered`. Чистый флаг + Б.1, нового нет.
3. **Ветка «бой»** → обычный энкаунтер: Тэо — `NonActingNpc` за линией подопытных;
   `marken_fate` решается тем, бьёшь ли линию жертв (Б.1).
4. **Реактивные реплики в бою** → переиспользуем `flavor` у фаз (текст на порогах HP).
   Пара барков Тэо — без нового кода.

**Итого новое = один примитив (story-choice → flag)**, и тот обязателен для эпилога.
Остальное — reuse (NonActingNpc, phase flavor, Б.1). Архитектурного unknown нет.

> **Уточнение по факту реализации (Волна 2).** Оценка «один примитив» оказалась
> неполной. Story-choice из Волны 1 (Атом 2) выставляет флаг и гейтит **строки
> диалога** (`DialogueLine.requires_flag`), но НЕ управляет потоком сцен — он
> линеен (`scene_index += 1`, скип только invisible-сцен). Чтобы «переговоры →
> бой 3 пропускается», понадобились ещё **два** примитива (оба app-side, без
> SCHEMA-bump, реализованы в Волне 2):
> 1. **scene-level `requires_flag`** на `SceneDef` (`c32cbcd`) — сцена (в т.ч.
>    `Combat`) скипается как invisible, если её флаг не выставлен. Контракт: скип
>    `Combat`-сцены отбрасывает её `on_victory_flags`/objective-флаги, поэтому
>    ветка-замена обязана выставить нужные downstream-флаги сама (см.
>    `docs/architecture.md`).
> 2. **line-level `excludes_flag`** (`a3eff0b`) — негатив к `requires_flag`:
>    показать реплику, когда флаг ОТСУТСТВУЕТ. Нужен «else»-веткам без
>    позитивного флага (лодка потеряна = `excludes boat_saved`; Маркен одержим =
>    `requires theo_killed` + `excludes marken_killed`).

**Опционально (полировка, не база):** капитуляция Тэо **по ходу** боя (добили
подопытных → сдаётся) — через фазу `turn_limit`/`HpBelowPct` → `victory_override`,
тоже на существующих фазах, без in-combat выбора.

## Б.4 — 3-фазный босс «Контейнер»: расконтейнинг (бой 4) ✅

**Ростер:** Хорст (босс-переговорщик) + Контейнер (3-фазный) + 2 охранника +
накопитель (за спиной Контейнера).

### Лор-логика фаз (почему босс так себя ведёт)

Контейнер — трёхслойный артефакт: **инженерная оболочка** (Хорст: броня) +
**связанный Гость** (Тэо: магия) + **накопитель-кристалл** (питание). Фазы — это
**последовательный отказ удержания**: партия проламывает оболочку, наружу слой за
слоем лезет сырой Гость. Тезис главы: методично «приручённый» Договор под
давлением остаётся неконтролируемым Гостем.

| Фаза | Лор | Кит | Приоритет партии |
|---|---|---|---|
| 1 «Изделие» (танк) | оболочка цела, ведёт себя как продукт | высокая броня, **низкий урон** мили | нужен pierce (Лира) / проломить |
| 2 «Гость проступает» (маг) | оболочка треснула, природа Гостя выражается | броня частично сбита, `spell_damage` (pierce) + статус-контроль | прерывать/бёрстить кастера |
| 3 «Несобранное» (подвижный) | оболочка разрушена, сырой нестабильный Гость | броня 0, высокая speed, дальнобой | догонять / дальнобоем |

Кит синтезирован из виденного игроком: искра (ch2) → дальнобой ф.3; симбионт-реген
(бой 2) → саморемонт ф.1; Гость-в-носителе → магия ф.2.

### Движок — всё на готовом

> **Плейтест-числа (реализовано).** Контейнер — **единый пул HP**, переносимый
> сквозь все 3 swap-фазы (без `heal_to_full`): непрерывное проламывание от полного
> до нуля. **HP = 60** (изначально 30 — удвоено после плейтеста: бой 4 кончался
> ~вдвое быстрее задуманного). Пороги фаз — `HpBelowPct`: **66 %** (свап в мага,
> ~40 HP) и **33 %** (свап в подвижного, ~20 HP).

- **Фазы** = swap-шаблоны на порогах HP (`HpBelowPct`), как 2-фазный босс ch1, но ×3.
  Статы/броня/способности/speed/теги — поля шаблона. **Ноль нового кода** (теги — Атом 3).
- **«Честный кайтинг» фазы 3 — отброшен.** Подвижность = swap-шаблон (броня 0 +
  high speed + дальнобой) под `Default` (он умеет reposition). Новый `AiBehaviorKind`
  **не нужен**.
- **Накопитель (хук «выбей ядро»)** = отдельный неподвижный юнит с HP за спиной
  босса; эмитит **баф-ауру** на Контейнер (`armor_bonus` + флейвор-питание).
  Выбил → аура пропала → босс уязвимее. **Аура — механизм ch2, ноль нового кода.**
  Тактика — callback к бою 2 («снять саппорт, а не дубасить танка»).
- **Обрыв ауры в фазе 3 «в любом случае»** = через **теги** (Атом 3): `aura.affects_tags = {symbiote}`;
  у Контейнера ф.1-2 есть тег `symbiote` → аура действует; свап в фазу 3 **снимает
  `symbiote`+`corporeal`, ставит `incorporeal`** (`Effect::RemoveTag`/`AddTag`) → аура не видит
  цель → отрубается гарантированно. Бонус: `incorporeal` → физ-атаки слабеют. Лор-native
  (перестал быть механизмом → питание отрезано).
- Хорст (торгуется/сдаётся) — `victory_override` фазы или `NonActingNpc`.

### Дельта реализации (общая с Б.2 — Атом 3)

Tag-предикат — один примитив, два потребителя: `requires_tags` хила (Б.2) и `affects_tags`
ауры (Б.4). При реализации (`engine-tz.md` Атом 3):
1. `aura_targets` вызывать из **обоих** aura-call-site (`aura_effects_on` + `aura_membership_set`);
2. фаза-3 мутирует теги через `Effect::AddTag/RemoveTag` (теги мутабельны → входят в `post_state_hash`).

Расу «симбионт» фаз 1-2 можно взять **ту же**, что у симбионтов боя 2 → значение
фильтра общее для хила пастуха и ауры накопителя.

## Б.5 — Партийный спутник Орен + ранение Aldric (бой 1→2) ✅

Орен — **4-й (временный) спутник**, виридиец-лекарь, **чистый саппорт**. Входит после
боя 1 (`party_add`), идёт до конца главы. Повествование — `description.md` §2 / Вставка 1;
характер — `docs/characters/oren.md`.

### Статы и кит (числа обоснованы через Лиру)

Lyra-референс (mage): HP 12, INT +3, mana 10, посох sp 1; `heal` = 2 маны → 1d4+3+1 ≈ **6.5**.

**Статы Орена:** HP 12, STR −1, INT +3 (=Лира), speed 3, **броня 1** (кольчуга, средняя),
фокус «живой ланцет» (1d4, sp 1), **mana 15**, **AP 1**.

| Способность | Цель | Эффект | Цена | Длит. | Движок |
|---|---|---|---|---|---|
| Удар ланцетом | enemy | 1d4−1 ≈ 1.5 | — | — | `weapon_attack` (есть) |
| Сшивание (хил) | ally | 1d4+3+1 ≈ 6.5 | 2 маны | — | `heal` (есть) |
| **Вливание жизни** (HoT) | ally | **+4/тик ×2 = +8** | 2 маны | 2 хода | ⚠️ **новый атом** |

Различие Сшивание↔Вливание (оба 2 маны): хил **мгновенный** (спасти сейчас), HoT — **+8 за
2 хода** (эффективнее, если цель доживёт). Burst-vs-efficiency.

### Движок — два новых бита

1. ⚠️ **Heal-over-time статус** (Вливание жизни). Статусы сейчас умеют только **damage-DoT**
   (`dot_per_tick` → `TickDot`). Нужен зеркальный heal-тик: поле `heal_per_tick`
   (**фиксированное, не INT-скейл** — как `dot_per_tick`) + эффект `TickHeal`. Маленькая,
   симметричная DoT добавка. *(Тот самый regen-атом, что отбросили в Б.2 для пастуха — здесь
   возвращается ради Орена.)*
2. **Хук «стартовый статус на партийца в энкаунтере»** — для `injured` на Aldric в бою 2
   (**−1 броня, −1 скорость**, затухает после боя 2). Партийцы спавнятся из party-стейта без
   авторских статусов → нужен способ навесить статус названному партийцу на энкаунтер (либо
   стартовый-статус-в-энкаунтере, либо флаг `injured` → применяется в сетапе боя 2).

Провокацию Aldric **не трогаем**: `injured(armor_bonus −1)` сам ослабляет её стойку через
стэк брони (`defending` + injury), Орен компенсирует баффом `armor_bonus +2` ценой действия.

Всё прочее (heal, `armor_bonus`, `speed_bonus −`, пул маны, `weapon_attack`) — существующее.

### Баланс — ребаланс боёв под четверых

Орен делает партию **вчетвером**; числа Б.2 считались под троих. Но Орен **урона почти не
даёт** → дамагеров остаётся ~3 (Aldric ослаблен `injured`), он добавляет sustain+контроль.
Ребаланс **мягче**, чем от 4-го DPS: «твой лекарь vs их пастух» — симметрия, усиливающая урок
боя 2. Мана 15 / 1 AP → ~5-7 ходов саппорта при боях 5-8. **Плейтест обязателен.**
